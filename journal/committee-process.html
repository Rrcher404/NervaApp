The One-Run Protocol

How a committee of six argumentative agents, a hard 8-out-of-10 gate, and a willingness to halt built a research app — and caught the bugs a tired solo builder would have shipped.

Built · graded · shipped

What this is

Most AI coding looks like this: ask, receive, hope. The One-Run Protocol replaces hope with a disagreement engine. Every unit of work is built by one agent, then attacked and graded by five others who are each built to notice — and refuse — different things. A composite score of 8.0 / 10 clears the work to proceed. Below that, it reworks. Twice. Then it halts and hands you a note.

The point is not the number. The point is that "is it good?" is a question no one can answer, and "does the capture survive airplane mode, does a private IP get through the scraper, does iOS lose the recording?" are questions five specialists can answer in parallel — and did.

BUILD (Okonkwo) one item, on a branch ↓ SELF-TEST (Kowalczyk) typecheck · unit · property · Playwright (headless) + the SAME flows in a REAL browser, screenshotted + a regression sweep of every prior item ↓ GATE (Solano convenes) five dimensions, each /10: 1 acceptance criterion, proven in a real browser (Kowalczyk) 2 worst-day UX — 11:40pm, ashamed, 4% battery (Kowalczyk + Halvorsen) 3 robustness — what the red team's attack found (Voss) 4 interface hospitality (Halvorsen) 5 constitution + banned-mechanics compliance (Marchetti) ↓ composite ≥ 8.0 → PROCEED · a veto or any data-loss finding → FAIL < 8.0 → REWORK (max ×2) · 2 failed regrades → HALT + handoff note

The crew

Personas are substrate, not costume. Each one's texture changes what it notices, not just how it talks. That is the whole trick — five identical reviewers give you one opinion five times; five different refusals stop the blind spots from overlapping.

How it actually helped — four cases from this build

Not hypotheticals. Every one of these was a real defect that a "looks done" pass would have shipped, each caught by exactly one specialist's narrow, hostile question.

Voss · dimension 3 · initial gate

The sacred write had no failure path

The one write the whole app calls sacred — the local save that must survive every API being down — was wrapped in try { … } finally { … } with no catch. A rejected IndexedDB write (quota, Safari private mode) lost the capture silently. Every test passed, because a missing catch block has no behaviour to exercise. Voss found it by attacking; it was obvious on reading. Fix: caught, loud, the input never cleared until the write lands. And: the Nakamura seat was created so reading-the-diff is somebody's actual job next time.

Kowalczyk · dimension 2 · rework cycle 2 → HALT

The fix for one bug shipped a worse one

To close a minor "duplicate capture across tabs" finding, rework added a 2-second content-dedupe. It could not tell a deliberate repeat from a double-click: type "wait", submit, pause a second, type "wait" again — the second thought vanished, and the success stamp fired over the discarded text. Two failed regrades in a row → the run halted itself and wrote a handoff note, exactly as designed. The dedupe was later deleted outright: it was a heuristic answering an identity question, and a dropped capture is a constitutional violation where a duplicate is a shrug.

Voss · dimension 3 · the harder pass

A live SSRF bypass through a canonicalised IPv6 address

The link scraper had a two-layer guard against fetching private addresses. Voss got http://[::ffff:127.0.0.1]/ straight through to an internal test server — because the URL parser rewrites that into [::ffff:7f00:1] before any guard sees it, so a regex against the dotted form never matched and the whole ::ffff:0:0/96 range walked in. Fix: the check now parses IPv6 numerically — mapped ranges, NAT64, unique-local, link-local — and a 25-payload torture battery could not beat the replacement.

Kowalczyk's cross-engine suite · item 2

An iOS bug hiding behind a green Chromium run

Voice capture stored the recording as a Blob in IndexedDB. Chromium: green. WebKit (real iOS Safari): put failed — WebKit can't persist Blobs in IndexedDB, which would have broken voice on the primary mobile target while every desktop test passed. Because the harness runs on two real engines, it surfaced immediately. Fix: store an ArrayBuffer, reconstruct the Blob only at upload.

The honest limit

The committee is rigorous, but it is still the app grading its own homework.

It substitutes for the builder during a run — the tripwire an ADHD solo founder cannot be for herself at 2am. It does not substitute for a human witness, or for the ten strangers whose wallets are the only real validation. It stopped a real data-loss bug from reaching a stranger today. That is the entire point. But the first catch still has to be logged in front of someone who was never in the room.

What to steal from this

  • Diversity over redundancy. Five different refusals beat five identical reviews. Assign the lenses, not the verdict.
  • A guard with no failing test is decoration. Break it on purpose, watch it catch, restore. Everything here was mutation-tested that way.
  • Review the fix harder than the feature. Every serious bug this run was introduced by a repair, not the original build.
  • Let it halt. The 2-rework cap is what keeps "autonomous" from becoming "expensively stuck." The halt is the safety feature.
  • Test on the engines your users actually hold. The iOS bug was invisible on desktop.
